# 02. n8n в этой архитектуре

## 2.1. Где хостить

n8n — оркестратор: он держит крон-расписания, длинные потоки и собственную БД
с историей запусков. **Не размещаем на Vercel** (serverless эфемерный, ноды
с состоянием не выживут).

Рекомендации:

| Вариант                | Плюсы                                | Минусы                              |
|-----------------------|--------------------------------------|-------------------------------------|
| **Hetzner CX22 + Docker Compose** | дёшево (~€4/мес), полный контроль | надо самому ставить TLS, бэкапы |
| **Render Background Worker**     | one-click, ssl автоматом         | дороже на нагрузке               |
| **Railway**           | удобный UI, бесплатный лимит         | региональная задержка, GDPR-вопросы |
| **VPS в РФ (Selectel/Timeweb)** | минимальный лаг до Telegram/MAX серверов | мониторинг и обновления своими силами |

Минимальная установка через Docker Compose:

```yaml
# docker-compose.yml на VPS
services:
  n8n:
    image: n8nio/n8n:latest
    restart: unless-stopped
    environment:
      - N8N_HOST=n8n.your-domain.ru
      - N8N_PROTOCOL=https
      - N8N_PORT=5678
      - WEBHOOK_URL=https://n8n.your-domain.ru/
      - GENERIC_TIMEZONE=Asia/Omsk
      - DB_TYPE=postgresdb
      - DB_POSTGRESDB_HOST=db.<ref>.supabase.co
      - DB_POSTGRESDB_PORT=5432
      - DB_POSTGRESDB_USER=postgres
      - DB_POSTGRESDB_PASSWORD=${SUPABASE_DB_PASSWORD}
      - DB_POSTGRESDB_DATABASE=postgres
      - DB_POSTGRESDB_SCHEMA=n8n          # отдельная схема, не public
      - PREMIUM_INBOUND_SECRET=${PREMIUM_INBOUND_SECRET}
      - TELEGRAM_OWNER_CHAT_ID=${TELEGRAM_OWNER_CHAT_ID}
      - MAX_BOT_TOKEN=${MAX_BOT_TOKEN}
      - MAX_API_URL=https://botapi.max.ru
    volumes:
      - n8n_data:/home/node/.n8n
    ports: ["5678:5678"]
volumes: { n8n_data: {} }
```

Поверх — Caddy/Nginx для TLS. Дальше — `Settings → Import workflow` из
`n8n/workflows/`.

## 2.2. Подключение к Supabase

Три варианта; обычно используем все три на разных нодах:

### A. Postgres node (предпочтительно для cron-выборок)

Credential: тип `Postgres`, поля:

- Host: `db.<project-ref>.supabase.co`
- Port: `5432`
- Database: `postgres`
- User: `postgres`
- Password: значение `SUPABASE_DB_PASSWORD`
- SSL: `Allow / Require` (Supabase требует TLS)

### B. HTTP Request node (PostgREST через Supabase API)

URL: `https://<ref>.supabase.co/rest/v1/leads?select=*&service_kind=eq.lawn_mowing`

Headers:

- `apikey: SUPABASE_SERVICE_ROLE_KEY`
- `Authorization: Bearer SUPABASE_SERVICE_ROLE_KEY`
- `Content-Type: application/json`
- `Prefer: return=representation` (для insert/update)

### C. HTTP Request node + RPC

Все «бизнесовые» функции в схеме (`upsert_contact_by_identity`, `create_lead`,
`segment_for_recurring_lawn_mowing` и т.п.) доступны через
`POST https://<ref>.supabase.co/rest/v1/rpc/<function_name>` с тем же набором
заголовков и body — JSON с параметрами `p_*`.

## 2.3. Основные воркфлоу

### 2.3.1. `01-lead-events.json` — приём событий

```
[Webhook /lead-events]
       │
       ▼
[IF: header x-premium-secret == $env.PREMIUM_INBOUND_SECRET]   ── 401 если нет
       │ ok
       ▼
[IF: body.type == 'lead.created']
       │
       ▼
[Postgres: select … from leads join contacts join contact_identities where l.id = leadId]
       │
       ▼
[Set: ownerText = «🆕 Новый лид …»]
       │
       ├─► [Telegram (notify owner)]   chat_id = $env.TELEGRAM_OWNER_CHAT_ID
       └─► [Postgres: insert into events ('owner.notified', …)]
       │
       ▼
[Respond 200]
```

Ноды и поля:

- **Webhook** (`n8n-nodes-base.webhook`): method=POST, path=`lead-events`,
  Response Mode=`Last node` → отдельным `Respond to Webhook`.
- **IF** (`n8n-nodes-base.if`): сравнение строк header против env.
- **Postgres** (`executeQuery`): SQL с `'{{$json.body.leadId}}'` в шаблоне.
- **Set**: формирует текст сообщения.
- **Telegram**: Operation=`Send Message`, ChatID=expression, parse_mode=HTML.

### 2.3.2. `02-recurring-mowing.json` — повторный покос

```
[Cron Mon/Wed/Fri 09:00]
       │
       ▼
[Postgres: select * from segment_for_recurring_lawn_mowing(12, 'recurring_mowing', null)]
       │
       ▼
[SplitInBatches batchSize=20]
       │
       ▼
[Wait 1s]
       │
       ▼
[IF: channel == 'telegram']
       ├─► [Telegram: send → external_id]
       └─► [HTTP Request: POST $MAX_API_URL/messages?...&user_id={{$json.external_id}}]
       │
       ▼
[Postgres: insert into campaign_recipients on conflict update]
```

Сегмент уже отфильтрован в SQL: не отписаны, согласие есть, последний покос > N
дней, в сезон, ещё не получали эту кампанию в последние 20 дней.

### 2.3.3. `03-seasonal-pool.json` — сезонная рассылка по бассейнам

То же, но cron `0 10 1,15 5 *` (1 и 15 мая, 10:00) и
`segment_for_seasonal_service('pool_cleaning', 'seasonal_pool_open', null)`.

### 2.3.4. `04-error-watch.json` — мониторинг

Каждые 5 минут читает `app_logs.level='error'` за последние 6 минут и шлёт
алерт в TELEGRAM_OWNER_CHAT_ID.

## 2.5. Уведомление о новых лидах

Vercel-функция (`api/telegram/webhook.ts` или `api/max/webhook.ts`) после
успешного `createLead()` шлёт `POST` на `N8N_WEBHOOK_URL` с телом:

```json
{
  "type": "lead.created",
  "leadId": "uuid",
  "contactId": "uuid",
  "serviceKind": "lawn_mowing",
  "channel": "telegram"
}
```

n8n: подгружает остальное из БД (имя, телефон, район…) и шлёт владельцу.
Так логика «как уведомлять» не зашита в Vercel и легко меняется.
